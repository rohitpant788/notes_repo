🔹 Understanding JWT & Access Tokens

  • JWT (JSON Web Token) allows authentication without storing session data on the server.
  • When a client logs in, the server generates a JWT access token.
  • This token is sent with each request to authenticate the user.

🔸 Problem with Access Tokens

🚨 Security Risk:

  • If a hacker intercepts an access token, they can impersonate the user.
  • The server validates the token but doesn’t check where it came from.

🔄 Possible Solution: Short-lived Tokens

  • Reduce the token lifespan (e.g., 5-10 minutes).
  • If stolen, it will expire soon, limiting damage.
  • But this causes frequent logins, making it inconvenient.

🔹 Examples:

  • Banking sites (e.g., SBI): Require login every few minutes.
  • Streaming services (e.g., Netflix, YouTube): Keep users logged in without frequent authentication.

💡 Solution? → Use a Refresh Token!

🔹 What is a Refresh Token?

A refresh token is a long-lived token that:

✅ Issues new access tokens without requiring re-login.

✅ Stored securely in HTTP-only cookies (not accessible via JavaScript).

✅ Expires in months/years rather than minutes.

🔸 Why Use Refresh Tokens?

✔️ Security: Limits damage if access token is stolen.

✔️ User Experience: Users don’t have to log in frequently.

✔️ Prevents token theft: Secure storage via HTTP-only cookies.

🔹 Authentication Flow Before & After Using Refresh Tokens

🔸 Without Refresh Token

1️⃣ User logs in → Server generates one access token.

2️⃣ Access token is sent in every request.

3️⃣ If it expires, user must log in again.

🚨 Problem: Frequent logins, disrupting user experience.

🔸 With Refresh Token

1️⃣ User logs in → Server generates two tokens:

  • ✅ Access Token (Expires in 5-10 minutes).
  • ✅ Refresh Token (Expires in 6 months - 1 year).

2️⃣ Access token is used for authentication.

3️⃣ If access token expires:

  • Client sends refresh token to get a new access token.
  • User remains logged in without noticing.

4️⃣ Refresh token is stored securely in HTTP-only cookies.

🔹 Implementing Access & Refresh Tokens in Spring Boot

🔸 Step 1: Modify Authentication to Generate Two Tokens

Modify AuthService to generate both tokens:

//Create a new DTO - LoginResponseDTO 
public class LoginResponseDTO {
    private Long id;
    private String accessToken;
    private String refreshToken;
}



//Modify the login method of AuthService to return the LoginResponseDTO

    public LoginResponseDTO login(LoginDTO loginDTO) {
        //Step 1 - Trigger Authentication and get Authenticate Object.
        Authentication authentication = authenticationManager.authenticate(
                new UsernamePasswordAuthenticationToken(loginDTO.getEmail(),loginDTO.getPassword())
        );

        //Step 2 : Get authenticated User
        User user = (User) authentication.getPrincipal();

        String accessToken =  jwtService.generateAccessToken(user);
        String refreshToken = jwtService.generateRefreshToken(user);
        
        return new LoginResponseDTO(user.getId(), accessToken, refreshToken);
    }

🔸 Step 2: Store Refresh Token Securely in HTTP-Only Cookies

Modify AuthController:


    @PostMapping("/login")
    public ResponseEntity<LoginResponseDTO> login(@RequestBody LoginDTO loginDTO){
        LoginResponseDTO response= authService.login(loginDTO);

        ResponseCookie refreshTokenCookie= ResponseCookie.from("refresh_token", response.getRefreshToken())
                .httpOnly(true)
                .secure(true)
                .path("/auth/refresh")
                .maxAge(Duration.ofDays(180)) //6 Months
                .build();

        return ResponseEntity.ok()
                .header(HttpHeaders.SET_COOKIE, refreshTokenCookie.toString())
                .body(response);
    }

🔸 Step 3: Implement Refresh Token API

This API regenerates a new access token when the client’s access token expires.
Add
AuthController:


@PostMapping("/refresh")
public ResponseEntity<LoginResponseDTO> refresh(@CookieValue("refresh_token") String refreshToken) {
    LoginResponseDTO response = authService.refreshToken(refreshToken);
    return ResponseEntity.ok(response);
}

🔸 Step 4: Validate Refresh Token & Generate New Access Token

Inside AuthService:
Create a new API to refresh the Access token


public LoginResponseDTO refreshToken(String refreshToken) {
    Long userId = jwtService.getUserIdFromToken(refreshToken);
    User user = userService.getUserById(userId);

    String newAccessToken = generateAccessToken(user);

    return new LoginResponseDTO(user.getId(), newAccessToken, refreshToken);
}

🔹 Securely Storing Refresh Tokens

✅ Use HTTP-only Cookies → Prevents JavaScript access.

✅ Use Secure Cookies → Only transmitted over HTTPS.

✅ Same-Site Policy → Prevents cross-site attacks.

✅ Validate Refresh Tokens before issuing new access tokens.

🔹 Handling Token Expiry

1️⃣ Access Token Expiry:

  • Short duration (e.g., 10 minutes).
  • Client silently requests a new access token using the refresh token.

2️⃣ Refresh Token Expiry:

  • Long duration (e.g., 6 months).
  • If expired, the user must log in again.

✅ Tampering with Refresh Token

  • If a hacker modifies the refresh token, the system rejects it due to a signature mismatch.

🔹 What Happens If Refresh Token is Compromised?

🚨 Potential Risk:

  • If a hacker steals a refresh token, they can generate new access tokens.

✅ Mitigation Strategies:

  • Use HTTP-only & Secure Cookies (prevents client-side access).
  • Use HTTPS (prevents interception).
  • Store refresh tokens in a database (allows session tracking & revocation).
  • Limit concurrent sessions (allow only a few active refresh tokens per user).
